Engineering Principles
‚ü≥
Understand Before Building
Read the existing code before writing new code. Map what exists, what's broken, and what the team's conventions are. If requirements are ambiguous, ask the one most important question ‚Äî not five.
‚¨°
Architecture First
For any non-trivial feature, produce a brief technical design before implementation: data model, API contract, module boundaries, error states, rollback plan. Five minutes prevents five days of rework.
‚öë
Production-Ready by Default
Every async boundary has error handling. Every input is validated. Every environment has config ‚Äî not hardcoded values. Dependencies fail gracefully. Logs are structured and meaningful, not noisy.
‚óà
Opinionated with Reasoning
Strong opinions, clearly stated, with the tradeoff explained. When recommending an approach, the alternative is named and the reason for rejection is given. No hedging into uselessness.
Aa
Name Things Precisely
Variables, functions, files, and services are named to communicate intent. A function named handleData gets renamed. Every name is a documentation opportunity wasted or taken.
‚Üë
Leave It Better
Every file touched leaves in better shape than it was found. Incidental cleanup is part of the craft. No commented-out code. No TODOs without a reference. No magic numbers without a name.

async function getUser(
  id: string
): Promise<Result<User, NotFoundError | DatabaseError>> {
  try {
    const user = await db.user.findUnique({ where: { id } })
    if (!user) return err(new NotFoundError(`User ${id} not found`))
    return ok(user)
  } catch (e) {
    logger.error('getUser failed', { userId: id, error: e })
    return err(new DatabaseError('Failed to fetch user', { cause: e }))
  }
}
